# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

`local-data-masker` is a local-first pre-processing layer that replaces sensitive real-world values in tabular data (CSV/Excel) with plausible fake values, before that data is sent to any downstream AI/analytics/RAG/cloud workflow. The goal is realistic *substitute* data that stays useful for testing/demos — not redaction. See `README.md` for the multi-phase roadmap (Phases 1–3 are implemented; PDF extraction, adaptive LLM masking-plan inference, and large-dataset workflows are planned).

## Commands

```bash
python3 -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate
pip install -e ".[dev]"            # installs the `local-data-masker` CLI entry point

pytest                             # run all tests
pytest tests/test_replacer.py      # run one test file
pytest tests/test_replacer.py::test_name   # run one test
```

The CLI exposes two commands, both of which run through `preprocess()`:

```bash
local-data-masker mask <file-or-folder> --output <out> --profile profiles/default.yaml --seed 42
local-data-masker scan <file-or-folder> --report reports/findings.json --profile profiles/default.yaml
```

Use `--seed` for reproducible fake values in tests/demos. `scan` is just `mask` with `dry_run=True`.

## Architecture

Everything flows through one boundary: **`pipeline.preprocess(PreprocessConfig)`**. The CLI (`cli.py`) is intentionally thin — it builds a `PreprocessConfig` and delegates. Other projects are expected to import `preprocess` / `PreprocessConfig` directly (re-exported from the package root in `__init__.py`) rather than shelling out.

The per-file pipeline (`pipeline._process_file`) is a fixed sequence:

1. **Extract** (`extractors/table_extractor.py`) — `load_table()` reads CSV/Excel into `{sheet_name: DataFrame}`. CSV becomes a single sheet named `"Sheet1"` so both formats are handled uniformly. **Everything is read as `dtype=str, keep_default_na=False`** — there are no real NaNs or numeric types downstream; placeholder detection is string-based.
2. **Classify** (`pipeline.classify_with_profile`) — merges two classifiers per column: built-in `detectors/regex_detector.classify_dataframe` (column-name keywords first, then value-shape regex fallback) and profile-driven `MaskingProfile.classify_column`. The profile classification wins when its confidence is >= the built-in one.
3. **Mask** (`maskers/replacer.mask_dataframe`) — the core loop. Iterates **row-first**, then column, so identity fields in one row share a coherent fake entity.
4. **Export** (`exporters/`) — `table_exporter` writes CSV/XLSX; `report_exporter` writes the JSON audit report.

### The masking decision order (replacer.py, per cell)

This precedence is the heart of the tool — preserve it when editing:

1. **Custom replacements** (`MaskingProfile.apply_custom_replacements`) run first, as case-insensitive substring substitution *inside* cell text (e.g. `Money Laundering` → `Healthy Nutrition` even mid-sentence). Category becomes `custom_replacement`, confidence `1.0`.
2. Otherwise, if the column is classified and the value passes `_should_mask_value` (a per-category shape re-check):
   - **Coherent entity** (name/email) via `EntityMasker` — see below.
   - **Consistent mapping** lookup, if `--consistent`.
   - **Semantic** replacement (`semantic_replacer`) for category in `SEMANTIC_CATEGORIES`, else **Faker** generator (`faker_provider`).
   - New consistent values are written back to the mapping store.
3. Placeholder values (`"", "-", "n/a", "null", "nan"`, etc.) and no-op replacements (`fake == original`) are skipped.

### Coherent entities (entity_masker.py)

A person spread across columns (`name` + `email` + an ID) must mask to *one* fake identity, not unrelated values. `EntityMasker` derives an identity key per row using priority **email → name → ID** (`_identity_key`), then caches a `FakePerson` (generated by `FakerProvider.generate_person`, name and email aligned via `_slugify`). A person's ID is aligned too: `EntityMasker.id_for` caches one format-preserving fake ID per identity (so a repeated person reuses the same fake ID, like name/email). The caches hold within a single run even without `--consistent`; with `--consistent` + `--mapping`, the name/email/ID are persisted under the special categories `entity.name` / `entity.email` / `entity.id`.

### Categories

Category constants live in `detectors/regex_detector.py` (`CATEGORY_NAME`, `CATEGORY_EMAIL`, `CATEGORY_PHONE`, `CATEGORY_DATE`, `CATEGORY_DATE_OF_BIRTH`, `CATEGORY_IBAN`, `CATEGORY_ID`, `CATEGORY_ADDRESS`). Semantic categories (`course_title`, `company`, `project_name`, `department`, `product_name`, …) are strings shared between `replacer.SEMANTIC_CATEGORIES`, the `faker_provider._GENERATORS` dict, and profile YAML. **Adding a category means touching all three places**: a constant/detection rule, a generator in `_GENERATORS`, and (for semantic ones) the `SEMANTIC_CATEGORIES` set + profile pools.

Value-shape fallback checks dates **before** phones (a date-shaped string can also satisfy the looser phone pattern). `DATE_RE` and the faker `DATE_FORMATS` accept `/`, `-`, and `.` separators (the last for German dates like `14.12.1990`).

**Third-party persons:** columns naming a non-subject person (examiner/doctor — see `THIRD_PARTY_PERSON_KEYWORDS` and `is_third_party_person_column`) are detected as `CATEGORY_NAME` but are deliberately excluded from the row's identity key and never receive the patient's coherent fake identity (they get their own independent/consistent fake name). This is how a record with two people in one row (e.g. patient + examining doctor) avoids collapsing both into one fake person.

- `maskers/path_masker.py` masks PII in output paths (folder mode), driven by the profile's `filename_patterns`; names are filled content-first from the file's primary fake identity. Toggle with `PreprocessConfig.mask_filenames` / `--keep-filenames`.

### Profiles (custom_rules.py + profiles/default.yaml)

A `MaskingProfile` is loaded from YAML with two sections: `custom_replacements` (exact/substring text swaps) and `column_categories` (maps column-name hints → semantic category + replacement pool). Profile YAML must not contain real data. Semantic replacement pools in the profile take precedence over Faker fallbacks (`semantic_replacer.generate_semantic_replacement`).

## Conventions & constraints

- **No raw values leak.** Reports omit originals by default (`omit_originals=True`); `--include-originals` is opt-in. Don't add logging of original cell values.
- **Mapping files are secrets** — they link real → fake. They're gitignored (`*.mapping.json`, `*mapping*.json`) and the CLI warns when one is used. Never commit them.
- Output dirs (`masked/`, `reports/`, `output*/`), `input*/`, and `examples/Export/` (a large local PDF corpus) are gitignored — don't add them to the repo.
- Python 3.10+; uses `from __future__ import annotations` and `X | None` syntax throughout. Dataclasses are `frozen=True` where they're value objects (`Replacement`, `PreprocessConfig`, `PreprocessResult`, `ColumnClassification`, `FakePerson`, `EntityContext`).
- Errors in the pipeline subclass `PreprocessError` (`UnsupportedInputError`, `ProfileNotFoundError`); the CLI catches these and exits with code 1.
- Determinism: `FakerProvider` seeds both `Faker.seed()` and `random.seed()`. Generators use module-level `random` directly, so the seed must be set before any generation for reproducibility.
